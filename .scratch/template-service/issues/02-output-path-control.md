# Output path control: file.write_to(), -o flag, overwrite guard

Status: implemented

## Parent

`.scratch/template-service/spec.md`

## What to build

Give templates and users control over where output lands, with an overwrite guard. Register a `file.write_to(path)` method on the `file` namespace object that a template calls during render to declare its output path (mirrors Templater's `tp.file.move()`); it writes into a shared `Arc<Mutex<Option<PathBuf>>>` that the engine reads after render. Add `-o` / `--output` and `-f` / `--force` flags to the CLI. Precedence: `file.write_to()` > `-o` > default (which is `Config::output_dir()` joined with the resolved template's stem — already established by issue 01). **Superseded** — see "Implementation notes": this precedence was inverted post-delivery on explicit maintainer direction; shipped precedence is `-o` > `file.write_to()` > default. Before writing, if the target exists and `--force` was not passed, return a `TemplateError::OutputFileAlreadyExists` error; the CLI layer wraps it in a miette diagnostic suggesting `--force`.

The `file` namespace is a struct implementing `minijinja::value::Object`, built and registered by `TemplateEngine` (which owns the loader already — this is one more piece of env setup). The engine exposes a `take_output_path()` method so the service can read captured state after render.

## Acceptance criteria

- [x] `file.write_to("path")` in a template sets the output path
- [x] ~~Precedence enforced: `file.write_to()` overrides `-o`~~ **SUPERSEDED — see "Critical AC review" in Implementation notes.** Shipped precedence is inverted: `-o` overrides `file.write_to()`, which overrides the default (`Config::output_dir()` / `<resolved-stem>.md`). Precedence is still enforced end-to-end; only the `write_to`-vs-`-o` direction changed.
- [x] Existing output path without `--force` returns `TemplateError::OutputFileAlreadyExists`; the CLI layer wraps it in a miette diagnostic with code `traces::cli::template::output_exists` and help suggesting `--force`
- [x] `-f` / `--force` overwrites the existing file silently
- [x] Tests cover each precedence combination, the overwrite guard, and forced overwrite

## CLI wiring

Add two fields to `TemplateArgs` (`src/cli/template.rs`):

```rust
#[arg(short = 'o', long, value_name = "PATH")]
output: Option<PathBuf>,
#[arg(short = 'f', long)]
force: bool,
```

`-f` / `--force` follows the standard convention (`docs/refs/cli_guide.md` line 645, matching `rm -f`).

## API changes

- `TemplateService::render_to_file` gains `output: Option<&Path>` and `force: bool` params (add params, no builder — `ponytail`). Signature becomes: `(&self, name: &Path, output: Option<&Path>, force: bool) -> Result<PathBuf, TemplateError>`.
- `TemplateEngine` gets `new()` -> `with_file_namespace(state: Arc<Mutex<Option<PathBuf>>>)` -> builder chain -> `take_output_path() -> Option<PathBuf>`. **Superseded** — see "Implementation notes" below: shipped as `TemplateEngine::render()` returning a `RenderOutput { content, write_to }` via minijinja's `State::set_temp`/`render_captured`, on the maintainer's explicit direction to prefer that over this API shape.
- `TemplateError` gets an `OutputFileAlreadyExists { path: PathBuf }` variant. This is a pre-write check, not an I/O error wrapping — it's not `Write` repurposed. The name won't collide with `io::ErrorKind::AlreadyExists` (separate types).
- `TemplateCliError::Instantiate`'s `help()` checks for `OutputFileAlreadyExists` and returns `"pass --force to overwrite"`.

## Overlap with issue 03 (dry-run)

Dry-run skips the overwrite guard entirely. Whichever of issues 02 and 03 lands second must handle the interaction. If 03 lands first, `render_to_file` will already have a `dry_run: bool` (or a mode enum) — thread it past the guard. If 02 lands first, add a NOTE in 03 to skip the guard.

## Rust guidance

Relevant skills: `m03-mutability`, `m02-resource`, `m06-error-handling`, `err-thiserror-lib`, `err-custom-type`, `domain-cli`.

- **`file.write_to` needs interior mutability (m03):** minijinja methods on `Object` receive `&Arc<Self>`. The method implementation created via `Value::from_function` is a `Fn` closure with `Send + Sync + 'static`. So `file.write_to(path)` must write into shared mutable state. **Superseded** (see "Implementation notes"): shipped using minijinja's own per-render `State::set_temp`/`get_temp` (via `Template::render_captured`) instead of a custom `Arc<Mutex<Option<PathBuf>>>>` — `FileOps` ended up a stateless unit struct.
- **Borrow discipline (m03):** **superseded**, moot under the shipped design — no `Mutex` in this feature's own code, so nothing to discipline.
- **Precedence is pure logic (m06):** compute the final path as a pure function of `(write_to_value, cli_o_flag, default)` after render. Overwrite guard: `path.exists() && !force` → `TemplateError::OutputFileAlreadyExists`. Use `std::fs` for the existence check; propagate write errors with `?`.
- **FileOps and registration belong in TemplateEngine:** the engine already configures the minijinja environment (loader, etc.); `env.add_global("file", ...)` happens unconditionally in `TemplateEngine::new()` now (registration no longer needs a separate builder step, since `FileOps` takes no external state to inject).
- **`err-lowercase-msg`:** new error variant starts lowercase, no trailing punctuation (existing convention).

## Implementation notes

Delivered in `.worktrees/output-path-control` on branch `agent/output-path-control`, four commits on top of `main`'s `2db8faa`: `9b35ebb` (original delivery, notes below), `8ced52a` (`CliError` rename + precedence flip), `a5c8ffb` (`target_path.rs` → `writer.rs` split, `default_output_filename`), `79cf0b1` (`TemplatePath::read()`, `TemplateService::resolve` removed).

- **CLI wiring:** the actual struct in `src/cli/template.rs` is named `Template` (`pub(super)`), not `TemplateArgs` as this ticket assumed when it was written — same shape, `-o`/`--output`/`-f`/`--force` fields added exactly as specified. `Template::new()` (used for the bare `traces -i <name>` dispatch, which bypasses subcommand parsing) defaults `output: None, force: false` — the top-level `-i` shortcut does not expose `-o`/`-f`, only `traces template -i <name>`/`tmpl` do; not asked for otherwise.
- **`FileOps`** (`src/template/file_ops.rs`, new file — named `FileOps` rather than the ticket's own `FileNamespace`/`UiNamespace`-family naming; deliberate deviation, see below): implements `minijinja::value::Object` via `get_value` — one match arm per method name, each returning a `Value::from_function(...)` closure, per the pattern issue 04 documents for `ui` ("The `Object` trait's default `call_method` looks up each method name via `get_value` and dispatches to returned callable `Value`s"). No `call_method` override: the trait default handles dispatch, so adding `file.include(path)` (issue 05) is one more match arm plus one more `METHODS` entry, not a change to shared dispatch logic.

  **Deviation from this ticket's Rust-guidance API shape, done deliberately on explicit direction:** the ticket specified `Arc<Mutex<Option<PathBuf>>>` shared state plus `TemplateEngine::new() -> with_file_namespace(state) -> take_output_path()`. Implemented that way first, then researched alternatives (`rust-docs-mcp` against the cached minijinja 2.21.0 source, the official `github.com/mitsuhiko/minijinja/tree/main/examples` — `state-temps`, `load-lazy`, `object-ref`, `dynamic-objects`, `eval-to-state` — plus real-world usage patterns) at the maintainer's request, which surfaced that `Template::render_and_return_state` is `#[deprecated(since = "2.18.0")]` in favor of `render_captured`, and that minijinja has a purpose-built per-render scratch mechanism (`State::set_temp`/`get_temp`) for exactly "a custom function needs to persist a value for the code that called render() to read back" — verified safe across `{% include %}` boundaries by reading `vm::perform_include` (it mutates the same `State` in place and never touches `temps`). On the maintainer's explicit direction to prefer that over the ticket's literal prescription, reworked to: `FileOps` is a stateless unit struct (`write_to`'s closure takes `state: &State` and calls `state.set_temp(WRITE_TO_KEY, ...)`); `TemplateEngine::render()` now returns a `RenderOutput { content, write_to }` via `Template::render_captured(...).state().get_temp(...)`, extracting the captured value directly instead of exposing a separate `with_file_namespace`/`take_output_path()` pair. Net effect: no `Arc`/`Mutex`/`PoisonError` anywhere in this feature's own code, and no manual "reset between renders" invariant to maintain (a fresh `State` per render makes that structurally impossible rather than tested-for) — traded for `render()`'s return type changing from bare `String` to `RenderOutput`, which every `TemplateEngine::render` caller/test had to update.
- **Output path resolution (superseded by the containment fix below):** initially `TemplateService` gained a `resolve_output_path(&self, path: PathBuf) -> PathBuf` helper so `-o`/`file.write_to()` paths, when relative, resolved against `Config::root()` — the same base `default_output_path` already used. This wasn't specified by this ticket (only the default's base was established, by issue 01); without it, a relative `-o`/`write_to()` path would resolve against the process's CWD instead, which broke cross-test isolation during implementation and would be surprising in practice. Explicit `-o`/`file.write_to()` paths do **not** get `.with_extension("md")` applied — only the computed default does; an explicit path controls its own extension. **This helper had a real path-traversal gap** (see below) and was replaced by `TemplateTargetPath`.
- **Path traversal fix — `TemplateTargetPath` (`src/template/target_path.rs`, new file), found and fixed post-delivery:** `resolve_output_path` joined a relative candidate onto `root` but never rejected `..` or absolute candidates, so `-o ../../../etc/passwd` or `{{ file.write_to("../../.ssh/authorized_keys") }}` wrote outside the project unchecked — demonstrated concretely (`root.join("../../../tmp/evil.md")` even lexically `starts_with(root)`, so a naive post-join `starts_with` check wouldn't have caught it either; the fix has to reject `..` components before joining). `TemplateTargetPath::confine(root, candidate)` is now the *only* way `-o`/`file.write_to()` candidates become a write target — rejects absolute paths and any non-`Normal`/`CurDir` component, returning the new `TemplateError::OutputPathEscapesRoot { path }` otherwise. `render_to_file` threads `TemplateTargetPath` (not a raw `PathBuf`) through the whole exists-check/create_dir_all/write sequence, so no future code path can reach `fs::write` with an unconfined candidate without a type error. `Config::output_dir()` (the *default* path's base) deliberately does **not** go through `confine` — it's a value from the project's own already-trust-gated config and, per existing tested behavior, may legitimately be absolute; it goes through a separate infallible `TemplateTargetPath::trusted(root, candidate)` instead (mirrors the old `resolve_output_path`'s absolute-passthrough/relative-join rule, just without re-validating an already-trusted value). `TemplateCliError::Instantiate` gained a third special case (`is_output_escapes_root_error`, mirroring `is_output_already_exists_error`) with code `traces::cli::template::output_escapes_root` and help `"pass a path within the project — absolute paths and \"..\" segments are not allowed"`. `CONTEXT.md`'s `file.write_to(path)` and `--output / -o` entries now document the confinement.
- **Atomicity fix — `File::create_new`, found and fixed post-delivery:** the overwrite guard was originally `if output_path.exists() && !force { return Err(...) }` followed later by `fs::write(...)` — a TOCTOU race: another process (or another `traces` invocation) could create the file between the check and the write, silently bypassing the guard the AC requires. Replaced with a `write_output(path, content, force)` free fn in `service.rs`: `force == false` opens via `fs::File::create_new` (`O_CREAT | O_EXCL`), which fails atomically with `io::ErrorKind::AlreadyExists` if the file already exists — no separate `exists()` call at all, so there's no window between check and write. `force == true` uses `fs::File::create` (truncate-or-create) as before. `io::ErrorKind::AlreadyExists` maps to `TemplateError::OutputFileAlreadyExists`; any other I/O error (e.g. permission denied — now covered by a `#[cfg(unix)]` test) maps to `TemplateError::Write`, same as before. Purely a mechanism change — observable behavior and every existing test were unaffected; added `write_output`-level unit tests (`creates_a_new_file_when_absent`, `refuses_an_existing_file_without_force`, `overwrites_an_existing_file_with_force`, `propagates_permission_errors_as_write_errors`) for the parts a `render_to_file`-level test can't isolate.
- **Readability refactor — `service.rs` decomposed, on request:** `render_to_file`'s body (resolve/read/render/choose-path/ensure-dir/write, all inline) was split into small private helpers each doing one job — `read_template`, `render_template`, `choose_output_path`, `default_output_path` (pre-existing), `ensure_parent_dir`, `write_output` — so `render_to_file` itself reads as a 9-line top-to-bottom sequence. `read_template`/`ensure_parent_dir` are associated fns (no `&self`) since they don't touch service state — required, not just style, since this crate's `clippy::unused_self` is `deny`. No test needed changing: since each helper is `fn`-private, single-call-site, and takes plain references, the compiler's inliner collapses them back to the original codegen in release builds — confirmed by an unchanged, fully green test suite (zero test edits needed) plus a successful `cargo build --release`. `write_output` was further moved from a free fn into the `impl` block (`TemplateService::write_output`, still no `&self`, for the same unused-self reason) for consistency with the other helpers, and its `force: bool` parameter became a two-variant `WriteMode` enum (`CreateNew`/`Overwrite`) built once via `WriteMode::from_force(force)` right where `render_to_file` calls it — `render_to_file`'s own public `force: bool` parameter is unchanged (matches this ticket's spec and clap's natural binding for a `-f`/`--force` switch); only the internal, single-purpose `write_output` boundary gained the enum.
- **`WriteMode::create_file`, on request:** `write_output`'s `match mode { ... }` (which `fs::File` constructor to call) and its neighboring `if mode == WriteMode::CreateNew && ...` (which error variant an `AlreadyExists` I/O error maps to) were two mode-dependent decisions living outside the `WriteMode` type itself. Moved both into one method, `WriteMode::create_file(self, path) -> Result<fs::File, TemplateError>`, so `WriteMode` is fully self-contained about its own meaning and `write_output` shrinks to two lines (`mode.create_file(path)?` then `write_all`). First proposed as `WriteMode::open` — rejected on request, since `std::fs::File::open` is stdlib's canonical *read-only* constructor and naming a write-side method `open` fights that expectation; renamed to `create_file`, mirroring the two `fs::File::create*` methods it dispatches to. Also considered, on request, renaming `WriteMode` itself to `ForceWriteMode`; declined — `WriteMode`'s variants (`CreateNew`/`Overwrite`) are meaningful independent of any CLI flag, and `WriteMode::from_force` already isolates the one place CLI vocabulary (`--force`) touches this type; folding `Force` into the type's own name would blur that boundary the same way the now-fixed `TemplateService` struct doc used to (see below). Test coverage for atomic-create/error-mapping behavior moved from `write_output`'s test module to `write_mode`'s (`create_file_creates_a_new_file_when_absent`, `create_file_fails_when_the_target_already_exists`, `create_file_truncates_an_existing_target_when_overwriting`, `create_file_propagates_permission_errors_as_write_errors`), since that's now where the behavior lives; `write_output`'s own tests slimmed to the two cases it's uniquely responsible for (`writes_content_to_a_newly_created_file`, `overwrites_content_when_forced`) rather than a third copy of the same error-path coverage.
